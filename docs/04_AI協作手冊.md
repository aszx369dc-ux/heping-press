# 04 AI 協作手冊

這份文件是交接時最重要的一份。

如果您不懂程式，也可以和 AI 一起合作，但前提是要會下對指令、會看結果、不要盲目全收。

## 可以和哪些 AI 合作

### ChatGPT

適合：

- 幫忙整理需求
- 幫忙寫 README、教學文件
- 幫忙解釋錯誤訊息
- 幫忙看 `git diff`

### Gemini

適合：

- 快速問概念
- 比較不同做法
- 幫忙整理文字

### Codex

適合：

- 直接在專案裡修改檔案
- 嚴格限制只改某幾個檔案
- 輸出 `git diff`
- 先檢查再改

## 跟 AI 合作的核心原則

### 原則 1：一次只做一件事

不要一次叫 AI：

- 改書名
- 加故事
- 修封面
- 發網站

最好拆開做。

### 原則 2：一定要限制 AI 可以改哪些檔案

請明確說：

- 只修改 `data.js`
- 不要修改 `script.js`
- 不要修改 `index.html`
- 不要修改 `style.css`

### 原則 3：一定要請 AI 提供 `git diff`

不要只看 AI 說「我改好了」。

請一定補一句：

`完成後請提供 git diff`

### 原則 4：看不懂也要先檢查改了哪些檔案

至少先看：

- 有沒有改到不該改的檔案
- 有沒有改太多
- 有沒有碰到網站功能檔案

## 哪些情況不要直接接受 AI 修改

### 1. AI 改了你沒叫它改的檔案

例如你只說改 `data.js`，結果它還改了：

- `script.js`
- `style.css`
- `index.html`

### 2. AI 沒有提供 `git diff`

### 3. AI 一次改太多功能

### 4. AI 用猜的改重要頁碼

像下面這些不能亂猜：

- `startPage`
- `totalPages`
- 封面／封底路徑
- 故事順序

## 如何描述需求

最好包含這四件事：

1. 只改哪些檔案
2. 不准改哪些檔案
3. 要完成什麼
4. 完成後要給 `git diff`

## 如何貼 `git diff` 給 AI

做法：

1. 先執行 `git diff`
2. 複製輸出內容
3. 直接貼給 AI
4. 再問：
   - 有沒有問題？
   - 有沒有改太多？
   - 是否符合需求？

## 如何請 AI 檢查程式

可以直接這樣說：

```text
請用 code review 方式檢查這次修改。
優先找 bug、風險、行為改變、遺漏測試。
如果沒有問題，也請明確說明。
```

## 如何讓 AI 只修改指定檔案

```text
請只修改 script.js。
不要修改 data.js。
不要修改 index.html。
不要修改 style.css。
不要重寫整支檔案。
請做最小修改。
完成後提供 git diff。
```

## 跟 AI 合作時的安全提醒

- 不要直接接受看不懂的修改
- 不要在沒看 diff 的情況下 commit
- 不要把密碼、驗證碼、Token 貼給 AI
- 不要叫 AI 幫您做 `git reset --hard`
- 不要叫 AI 幫您 force push
